能看出来。你这个问题我给你一个先结论：

**你当时大概率不是“只能拿到 2011”**，而是 **“你的项目在方法上和工程上都被做成了 2011 全国基线版”**。因为 CHARLS 下载页里混在一起的，其实不是同一种数据：有的是**试调查**，有的是**全国正式波次**，有的是**补充专题调查**，有的是**二次整理后的 harmonized 数据**。而你现在这个仓库的 README 和路径配置，明确写的就是 `2011年协变量.csv` 和 `2011年FI+CVD及变量新.csv`，说明项目本身就是按 2011 一年做的，不是多波次模型。 

再结合官方说明，这个选择其实很合理：CHARLS 是 **2008 两省 pilot 试调查**起步，**2011/12 才是全国基线调查**，后面 **2013、2015、2018、2020** 才是全国样本的后续追踪；官网也说明数据通常会在采集结束后一段时间后才公开，所以如果你项目做得更早，后面的波次当时未必已经公开到你能直接用。([CHARLS][1])

你看到这些名字，最好这样理解：

**1）Pilot = 试调查 / 试点调查**
2008 CHARLS Pilot 官方中文页就写的是“**两省试调查**”；2011 全国基线用户手册也明确说，2008 只在 **甘肃和浙江** 两省做 pilot，用来验证 HRS 型调查在中国能不能跑通，不是全国代表性样本。2012 Pilot Wave2 则是这个 pilot 样本的第二轮追踪。也就是说，**2008 和 2012 这一对，是“两省样本线”**，不是你做全国 CAD 预测时最自然的首选。([CHARLS][2])

**2）Wave = 正式调查的“第几波”**
Wave 不是“随便起的名字”，而是纵向队列研究里很标准的说法，意思是**同一个正式调查体系中的第几轮访查**。CHARLS 官方手册写得很清楚：**2011/12 是 national baseline（全国基线）= Wave 1，2013 是 Wave 2，2015 是 Wave 3，2018 是 Wave 4，2020 是 Wave 5**。所以 Wave 1/2/3/4/5 才是全国正式主线。([CHARLS][1])

**3）Life History 不是普通 core wave，而是专题补充调查**
2014 Life History Survey 不是“全国追踪第几波”的同类替代物，而是一次**生命历程回顾调查**。官方 harmonized life history 文档明确说，这个调查主要收的是受访者**进入研究之前的人生经历**，比如婚育史、居住史、工作史、童年状况、父母和祖辈信息等，核心是**回顾性历史信息**，不是你理解的那种“某一年当下的常规体检+疾病状态”主表。([CHARLS][3])

**4）Harmonized CHARLS = 二次整理后的“研究友好版”**
这不是“另一波调查”，而是 Gateway to Global Aging 做的**整理版/统一版**。它的目的，是把多波次里常用、可比较的变量整理成研究更容易直接上手的格式。官方说明里说得很明确：harmonized 数据会把可比变量尽量统一命名、统一定义，并把多波次合在一套研究友好数据里；但它**不包含原始 CHARLS 的全部题目**，只保留“常用且可比较”的那部分。换句话说：**更好用，但不如原始全量数据全。**([全球老龄化数据门户][4])

所以你当时只用 2011，我看最可能有四个原因：

**第一，2011 是第一套全国基线，最干净、最自然。**
你做的是基于 FI 预测 CAD 风险的模型。对于这种项目，拿全国基线做一个“横断面基线模型”是最常见也最省事的起点；2008/2012 pilot 只有两省，不适合直接当全国泛化的数据底座。([CHARLS][1])

**第二，你仓库工程上就是按 2011 单波次搭的。**
README 直接写了数据说明是“2011年协变量.csv”和“2011年FI+CVD及变量.csv”；路径配置里也是把原始 covariates 和 targets 都写死为 2011 文件。也就是说，哪怕网站上有别的年份，你当时的代码工程也没有设计成“多波次读取—对齐—合并—建模”。 

**第三，你项目用到了实验室/生物标志物变量，而这些并不是每一波都齐。**
你仓库的变量配置里明确用了 `glucose_mgperdl`、`ldlCholesterol_mgperdl`、`cReactiveProtein_mgperdl`、`cystatinC_mgperl`、`hemoglobin_gperdl` 这类检验指标。官方 2015 blood release note 明确指出：**2013 的 Wave 2 follow-up did not include the collection of biomarkers**。这就意味着，如果你想把 2011 和 2013 直接并成同构特征集，你会马上撞上“关键实验室变量缺失”的问题。 ([CHARLS][5])

**第四，后续波次不是不能用，而是“不能无脑混用”。**
2014 Life History 是回顾模块；Harmonized 又是裁剪过的研究版；2013 又缺某些 biomarker；后面 2015/2018/2020 虽然是正式 wave，但你要做多波次模型，就必须重新处理**变量同名不同义、题目并非每波都问、化验并非每波都测、结局定义跨波一致性、ID 追踪和样本刷新**这些问题。官方 harmonized 说明也强调，它就是为了解决“多文件、多波次难合并”的麻烦而存在的。([全球老龄化数据门户][6])

所以你问“是不是咱们只能用 wave 名字的这种形式”，答案是：

**不是。不是只有 Wave 才能用，而是要看你的研究问题。**

* 你如果要做**全国正式纵向主线**，优先用 **Wave1/2/3/4/5**。([CHARLS][1])
* 你如果要做**方法学试验**、或者研究两省 pilot 样本本身，可以用 **2008 Pilot / 2012 Pilot Wave2**。([CHARLS][2])
* 你如果要研究**童年经历、工作史、婚育史、生命历程暴露**，那就要用 **2014 Life History**。([CHARLS][3])
* 你如果要做**跨波次、跨国家、快速上手**，可以先看 **Harmonized CHARLS**，但要接受它不是全题库。([全球老龄化数据门户][4])

如果把你这个旧项目放到今天重做，我会建议你分三条路线选一条：

**路线 A：复刻旧项目**
继续只用 **2011 全国基线**。这样最容易和你原仓库保持一致，也最容易复现实验。你现在仓库本身就是这么配的。

**路线 B：做真正的预测**
用 **2011 作为基线协变量**，然后把 **后续 wave 的 CAD/CVD 事件**作为随访结局，做纵向预测。这样比“同年横断面预测同年 CAD”更像真正预测。全国 wave 的时间线官方是完整的：2011/12 基线，2013、2015、2018、2020 follow-up。([CHARLS][7])

**路线 C：做多波次 pooled model**
把 2011、2015、2018、2020 里**共同存在**的变量抽出来建模；如果你坚持用实验室指标，那就要非常仔细核查每一波是否都有同口径变量，尤其别把 2013 直接混进去当成和 2011 一样。2013 follow-up 缺 biomarkers 这点已经有官方说明。([CHARLS][5])

一句话概括就是：

**你当年只用 2011，不像是“别的年份没有”，而更像是“2011 是第一套全国基线，又正好最适合你这个单波次、含化验指标的 CAD 预测项目”。**

### 3）什么叫“删失”

## 九、什么叫“调超参数”，你该怎么调

超参数不是模型自己学出来的参数，而是你**事先设定的旋钮**。

比如：

* 树要长多深
* 正则化要多强
* 学习率多大
* 少数类要不要加权

### 你最适合的调参方式

不要手调，不要凭感觉调。

直接用：

* **随机搜索** 或
* **网格搜索**

并且只在**训练集内部交叉验证**做。

### 一个很实用的思路

#### Logistic

* `penalty`: `l1 / l2`
* `C`: 0.01, 0.1, 1, 10
* `class_weight`: `None / balanced`

#### 随机森林

* `n_estimators`: 200, 500
* `max_depth`: 3, 5, 8, None
* `min_samples_leaf`: 1, 5, 10
* `class_weight`: `None / balanced`

#### XGBoost

* `learning_rate`: 0.01, 0.05, 0.1
* `max_depth`: 2, 3, 4, 5
* `subsample`: 0.6, 0.8, 1.0
* `colsample_bytree`: 0.6, 0.8, 1.0
* `reg_alpha`: 0, 0.1, 1
* `reg_lambda`: 1, 5, 10
* `scale_pos_weight`: 负类数 / 正类数 附近

类别不平衡这件事，scikit-learn 里的 `class_weight="balanced"` 和 XGBoost 的 `scale_pos_weight` 都是正规做法。前者会按类别频数自动算权重；后者官方文档直接说可按 `negative / positive` 的比例考虑。([scikit-learn.cn][9])

## 十、模型怎么评估，不能再看“准确率幻觉”

你之前“几乎都预测没病，但准确率看着还行”，这就是典型的**类别不平衡陷阱**。

如果 90% 的人都没发病，一个模型全预测 0，准确率都可能很高，但临床上一点用都没有。

### 这次至少看这些指标

* **AUROC**
* **AUPRC / AP**
* 敏感度 sensitivity
* 特异度 specificity
* PPV
* NPV
* **Brier score**
* **校准曲线**

为什么 AUPRC 很重要？
因为 scikit-learn 文档明确写了，Average Precision 是基于 precision-recall 曲线总结出来的，对二分类概率分数很适合，尤其在不平衡问题里很常用。([Scikit-learn][10])

为什么要看 Brier 和校准？
因为临床预测不是只看“排得准不准”，还要看“**0.2 的人是不是真的大约 20% 会发病**”。Brier score 衡量预测概率和真实结果的均方差，越小越好；校准曲线则是看预测概率和真实发生率是否一致。([Scikit-learn][11])

### 阈值别固定 0.5

0.5 不是神圣阈值。
你应该在验证集上决定阈值，比如：

* 更看重漏诊少：提高召回率
* 更看重误报少：提高精确率
* 想折中：选 Youden index 或 F1 最优阈值

# 一、为什么我不建议你继续把 PyCaret 当主控

你旧仓库里其实已经很明显是 **PyCaret + 多模型比较** 路线了：

* `compare_models` 选前 10 个
* 5 折 CV
* 按 AUC 排序
* `tune_model` 迭代 50 次
* 但 `fix_imbalance` 设成了 `False`，没有主动处理类别不平衡。

你保存下来的 PyCaret 模型配置里，确实也扫过很多模型：
Logistic、Ridge、LDA、RandomForest、ExtraTrees、GradientBoosting、LightGBM、AdaBoost、GaussianNB、QDA 等。

PyCaret 官方文档也说明了，它的 `compare_models()` 本质上是**把模型库里的估计器用交叉验证全跑一遍再排名**，`tune_model()` 则是对给定模型做超参数搜索。这个东西**做快速 baseline 很方便**，但它的问题是：
你一旦研究设计比较复杂，比如你现在这种“2011 基线 → 后续新发事件”的纵向任务，很多关键逻辑——**按人划分训练/测试、先分再填补、先分再特征选择、分 horizon、做删失处理、做概率校准**——你都更需要**显式写出来**，而不是藏在黑盒 setup 里。([PyCaret][1])

所以我对 PyCaret 的建议是：

**可以留，但降级成辅助角色。**
它适合：

* 先看看哪些模型大概有戏
* 做一个很快的 baseline 对照

它不适合：

* 作为你这次项目的核心训练引擎
* 承担最终论文主结果

# 二、这次为什么不建议神经网络先上

这点我给你一个非常明确的判断：

**第一阶段，不建议把神经网络当主力。**

原因不是“神经网络不高级”，而是**你的任务是典型中等规模表格数据的临床预测**。
NeurIPS 2022 的一篇很有代表性的基准论文做了大量超参搜索后，结论是：在**典型中等规模 tabular data（大约 1 万级样本）**上，树模型如 XGBoost、Random Forest 这类方法依然整体上优于深度学习方法。([NeurIPS 会议论文集][2])

而你这个 CHARLS 课题正好很像这种场景：

* 表格数据
* 混合变量
* 样本量不是百万级
* 缺失值多
* 变量定义复杂
* 临床解释性很重要

这正是树模型和线性模型的主场，不是神经网络最舒服的场。

所以我的建议是：

### 第一阶段

不用神经网络做主模型。

### 第二阶段

如果前面跑通了，你想“试试看”，可以加一个：

* MLP baseline
  或者
* AutoGluon 的更强 preset
  但它们是**加试题**，不是主线。

# 三、为什么这次要保留“多模型比较”，但不能乱比

你不是不能多模型比较。
你是**不能再像以前那样一股脑比一堆模型**。

这次正确的思路是：

## 只比“4+1”个模型家族

### 1）Logistic 回归：主解释模型

这个模型一定要保留。

因为在医学研究里，它有几个不可替代的优点：

* 最容易解释
* 最容易做列线图/风险解释
* 最适合当论文主 baseline
* 最容易看 FI 是不是真有独立预测作用

scikit-learn 官方文档里，`LogisticRegression` 支持 L1、L2、Elastic Net 等正则化方式；这很适合你这种变量不少、可能共线性也不弱的任务。([PyCaret][1])

### 2）XGBoost：主性能模型

这个基本可以定成你这次最重要的强模型之一。

XGBoost 官方参数文档里说明，二分类常用目标函数是 `binary:logistic`；对类别不平衡，`scale_pos_weight` 专门就是为正负样本失衡准备的，而且官方直接给了常用经验值：
`negative / positive`。([XGBoost 文档][3])

XGBoost 的优点：

* 非线性强
* 处理特征交互强
* 对表格数据很稳
* 社区最成熟
* 可解释性工具多（SHAP）

### 3）LightGBM：高效强模型

LightGBM 也是主力之一。

官方文档里，`LGBMClassifier` 支持 `num_leaves`、`max_depth`、`learning_rate`、`n_estimators`、`reg_alpha`、`reg_lambda` 等关键参数。对于类别不平衡，官方特别提示：
二分类更常用的是 `is_unbalance` 或 `scale_pos_weight`，而且这些做法**可能会让概率估计变差**，必要时应考虑做**概率校准**。([LightGBM][4])

这条对你很重要，因为你做的是**风险预测**，不是只要分类对错。

### 4）CatBoost：如果你保留原始分类变量，它会很好用

CatBoost 是这次我很建议你保留的模型。

原因是它对**分类变量**很友好。
官方文档明确说，CatBoost 支持直接指定 `cat_features`；而且它的分类特征文档甚至明确提醒：**不要在预处理时先 one-hot 编码**，因为那样会影响训练速度和质量。([catboost.ai][5])

所以如果你这次打算保留一条“原始分类变量分支”，CatBoost 很适合。

另外它对类别不平衡也支持：

* `class_weights`
* `auto_class_weights`
* `scale_pos_weight`。([catboost.ai][6])

### 5）随机森林：留一个就够

随机森林我建议留，但只作为**传统树模型基线**。

它的意义主要是：

* 训练稳
* 不太挑数据
* 可以对比“bagging 树”与“boosting 树”的差别

但一般来说，真正拼性能，XGBoost / LightGBM / CatBoost 往往更值得重点投入。
所以 RF 放在“保底与对照”位置最合适。([NeurIPS 会议论文集][2])

# 五、每个模型怎么调参，给你一个“小白版”

你现在不用追求“全参数宇宙搜索”，只要把关键旋钮抓住就行。

## 1）Logistic 回归

主要调：

* `penalty`：`l1 / l2 / elasticnet`
* `C`：正则强度的反比，越小惩罚越强
* `class_weight`：`None / balanced`

### 推荐搜索范围

* `penalty`: `l2`, `elasticnet`
* `C`: `0.01, 0.03, 0.1, 0.3, 1, 3, 10`
* `l1_ratio`: `0.1, 0.3, 0.5, 0.7, 0.9`（仅 elasticnet）

---

## 2）XGBoost

主要调：

* `learning_rate`
* `max_depth`
* `min_child_weight`
* `subsample`
* `colsample_bytree`
* `reg_alpha`
* `reg_lambda`
* `scale_pos_weight`
* `n_estimators`

### 推荐搜索范围

* `learning_rate`: `0.01 ~ 0.2`
* `max_depth`: `2 ~ 6`
* `min_child_weight`: `1 ~ 10`
* `subsample`: `0.6 ~ 1.0`
* `colsample_bytree`: `0.6 ~ 1.0`
* `reg_alpha`: `0 ~ 5`
* `reg_lambda`: `0.5 ~ 20`
* `scale_pos_weight`: 从 `负样本数 / 正样本数` 附近开始搜。官方就是这么建议的。([XGBoost 文档][3])

---

## 3）LightGBM

主要调：

* `num_leaves`
* `max_depth`
* `learning_rate`
* `min_child_samples`
* `subsample`
* `colsample_bytree`
* `reg_alpha`
* `reg_lambda`
* `n_estimators`
* `scale_pos_weight` / `is_unbalance`

### 推荐搜索范围

* `num_leaves`: `8 ~ 64`
* `max_depth`: `-1 或 3~8`
* `learning_rate`: `0.01 ~ 0.15`
* `min_child_samples`: `10 ~ 100`
* `subsample`: `0.6 ~ 1.0`
* `colsample_bytree`: `0.6 ~ 1.0`
* `reg_alpha`: `0 ~ 5`
* `reg_lambda`: `0 ~ 20`

LightGBM 官方还提醒了一个点：
如果你用了不平衡权重，**概率校准可能会变差**，所以后面最好加校准。([LightGBM][4])

---

## 4）CatBoost

主要调：

* `depth`
* `learning_rate`
* `l2_leaf_reg`
* `iterations`
* `subsample`
* `auto_class_weights` / `scale_pos_weight`

### 推荐搜索范围

* `depth`: `4 ~ 8`
* `learning_rate`: `0.01 ~ 0.15`
* `l2_leaf_reg`: `1 ~ 20`
* `iterations`: `200 ~ 2000`
* `auto_class_weights`: `None / Balanced / SqrtBalanced`

---

## 5）Random Forest

主要调：

* `n_estimators`
* `max_depth`
* `min_samples_split`
* `min_samples_leaf`
* `max_features`
* `class_weight`

### 推荐搜索范围

* `n_estimators`: `200 ~ 1000`
* `max_depth`: `3 ~ 15 或 None`
* `min_samples_leaf`: `1 ~ 20`
* `min_samples_split`: `2 ~ 20`
* `max_features`: `sqrt / log2 / 0.5`

# 六、调参工具到底用什么

## 正式推荐：Optuna

这是我最推荐你的。

Optuna 官方就是一个**框架无关**的超参数优化工具，可以定义 `objective`，然后 `study.optimize()` 自动搜索；官网也直接给了 XGBoost 和 LightGBM 的超参优化例子。([Optuna][7])

为什么 Optuna 比你直接乱用 AutoML 更合适：

* 搜索空间你自己定
* 目标指标你自己定
* 和 sklearn / xgboost / lightgbm / catboost 都很好接
* 容易控制“只在训练集内部 CV 调参”
* 比 PyCaret 黑盒更透明

### 如果你想让 AI/agent 直接帮你玩调参

Optuna 官方还有一个 **Optuna MCP Server**，官方 GitHub 仓库就叫 `optuna-mcp`，用途之一就是让 LLM 通过 MCP 和 Optuna 交互，做自动化优化和结果分析。([GitHub][8])

这个东西对你这种“想让 agent 帮忙跑调参”的场景，反而很对路。

## 正式推荐：MLflow Tracking

这个非常适合你。

所以：

# 八、如果你还想保留 AutoML，要选谁

## 1）FLAML：轻量、省时间

FLAML 官方文档里，`AutoML.fit()` 支持设置 `task`、`metric`、`time_budget`，非常适合先做一个便宜的自动化 baseline。([GitHub Microsoft][11])

适合你用来：

* 快速看看大概哪个模型家族靠谱
* 当成“低成本 baseline”

## 2）AutoGluon Tabular：强，但是更重

AutoGluon 官方文档里，Tabular 现在的 preset 里：

* `medium` 适合原型
* `best` 适合 serious usage
* `extreme` 会引入最近的 tabular foundation models（如 TabPFNv2、TabICL、Mitra、TabM），而且官方明确说 **best/extreme 更重，extreme 最好用 GPU**。([AutoGluon][12])

所以 AutoGluon 很强，但对你现在的阶段有两个问题：

* 重
* 更像“后期冲分工具”，不是第一阶段主线

### 结论

* **第一阶段主线**：自己写模型 + Optuna + MLflow
* **AutoML baseline**：FLAML 或 PyCaret
* **后期冲性能**：AutoGluon
